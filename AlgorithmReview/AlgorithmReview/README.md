# AlgorithmReview 

โปรเจกต์นี้เป็นการทบทวนโครงสร้างข้อมูลพื้นฐาน (Array, Stack, Queue) ด้วยภาษา Java

## 1. ArrayCaseStudy
**Time Complexity:** - การวนลูปเพื่อหาคะแนนรวม สูงสุด ต่ำสุด และนับจำนวน ใช้ Time Complexity เป็น **O(n)** โดยที่ n คือจำนวนสมาชิกใน Array
การวิเคราะห์ความซับซ้อนของเวลา (Time Complexity Analysis)
Time Complexity: O(n) โดยที่ n คือจำนวนสมาชิกใน Array (ในกรณีนี้ n = 10)

คำอธิบาย: โปรแกรมทำการวนลูป for เพียงรอบเดียวเพื่อเข้าถึงสมาชิกทุกตัวใน Array เพื่อคำนวณผลสะสมเปรียบเทียบค่าสูงสุด-ต่ำสุด และนับจำนวนตามเงื่อนไข ทำให้ระยะเวลาในการประมวลผลเติบโตเป็นเส้นตรงแปรผันตรงตามจำนวนข้อมูลนักศึกษา

**ผลลัพธ์การรันโปรแกรม:Array**
```text
=== ผลการวิเคราะห์คะแนนสอบ (Array) ===
1. คะแนนรวมของนักศึกษาทั้งหมด: 62
2. คะแนนเฉลี่ย: 6.2
3. คะแนนสูงสุด: 10
4. คะแนนต่ำสุด: 2
5. จำนวนนักศึกษาที่ได้คะแนนตั้งแต่ 7 คะแนนขึ้นไป: 5 คน
6. ลำดับนักศึกษาที่ควรได้รับการทบทวนเพิ่มเติม (คะแนน < 5): คนที่ 3 (ได้ 4 คะแนน) คนที่ 8 (ได้ 3 คะแนน) คนที่ 10 (ได้ 2 คะแนน)
```
## 2. StackCaseStudy
การทำงานของโปรแกรม
- นำเข้าคำสั่งทั้งหมดจัดเก็บลงในโครงสร้างข้อมูล Stack<String> ด้วยฟังก์ชัน push() ตามลำดับ
- แสดงรายการคำสั่งทั้งหมดที่รออยู่ในคลัง Stack ปัจจุบัน
- ดำเนินการจำลองการกด Undo จำนวน 2 ครั้ง ผ่านฟังก์ชัน pop()
- มีการตรวจสอบสถานะด้วยฟังก์ชัน isEmpty() ก่อนสั่งการ pop() ทุกครั้ง เพื่อป้องกันข้อผิดพลาดประเภท Stack Underflow
- แสดงคำสั่งที่โดนลบ (Undo) ออกไป และแสดงสถานะล่าสุดของ Stack หลังจากหักลบเสร็จสิ้น

อธิบายการทำงานตามหลักการ LIFO (Last-In, First-Out)
LIFO (เข้าทีหลังสุด ออกก่อนสุด): โครงสร้างข้อมูล Stack ทำงานเหมือนกองหนังสือ ข้อมูลที่เพิ่มเข้ามาเป็นลำดับสุดท้ายจะไปอยู่บนสุดของกอง จากสถานการณ์ข้างต้น คำสั่ง Type Java ถูกป้อนเข้ามาเป็นตัวสุดท้าย (push ตัวที่ 5) เมื่อผู้ใช้ต้องการกดยกเลิกการทำงานหรือสั่งถอนข้อมูลออก (pop) ตัวคำสั่ง Type Java ซึ่งอยู่บนสุดจึงเป็นข้อมูลชุดแรกที่หลุดออกจาก Stack ไปทันที

การวิเคราะห์ความซับซ้อนของเวลา (Time Complexity Analysis)
Time Complexity: O(1) สำหรับทุกฟังก์ชันพื้นฐาน
คำอธิบาย: การทำงานของคำสั่ง push(), pop(), และ isEmpty() ใน Java Stack จะเข้าถึงและจัดการข้อมูลเฉพาะที่บริเวณยอดบนสุด (Top of Stack) เท่านั้น โดยไม่ต้องเสียเวลาในการค้นหา ท่องไปในโครงสร้าง หรือขยับตำแหน่งสมาชิกตัวอื่น ทำให้ใช้เวลาในการประมวลผลคงที่ ไม่ขึ้นกับปริมาณข้อมูลในคลัง

**ผลลัพธ์การรันโปรแกรม:Stack**
```text
=== ระบบ Undo (Stack) ===
คำสั่งทั้งหมดใน Stack ปัจจุบัน: [Type Data, Type Structure, Delete Structure, Type Algorithm, Type Java]

ดำเนินการ Undo 2 ครั้ง...
Undo ครั้งที่ 1: ยกเลิกคำสั่ง [Type Java]
Undo ครั้งที่ 2: ยกเลิกคำสั่ง [Type Algorithm]

สถานะของ Stack หลังจาก Undo: [Type Data, Type Structure, Delete Structure]
```

## 3. QueueCaseStudy
**ผลลัพธ์การรันโปรแกรม:Queue**
อธิบายการทำงานตามหลักการ FIFO (First-In, First-Out)
FIFO (มาก่อน ได้ออกก่อน): โครงสร้างข้อมูล Queue มีพฤติกรรมเสมือนแถวคอยซื้อสินค้าทั่วไป คนที่เข้ามาต่อแถวเป็นคนแรกสุดจะอยู่หัวแถวและได้ออกจากแถวก่อน จากกรณีศึกษานี้ คนไข้ P001 และ P002 เข้ามาต่อแถวลงทะเบียนเป็นกลุ่มแรก เมื่อเจ้าหน้าที่เรียกคิว ทั้งสองคนจึงได้รับสิทธิ์เข้าห้องตรวจก่อนตามลำดับ ในขณะที่คนไข้มาใหม่ P006 และ P007 จะถูกส่งไปต่อแถวที่ปลายสุด (Tail) ของคิวเพื่อรอคอยตามคิวอย่างยุติธรรม

การวิเคราะห์ความซับซ้อนของเวลา (Time Complexity Analysis)
Time Complexity: O(1) เมื่ออิมพลีเมนต์ด้วย LinkedList

คำอธิบาย: การเพิ่มข้อมูลเข้าท้ายคิว (add()) การนำข้อมูลออกหน้าคิว (remove()) รวมไปถึงการแอบดูหัวคิว (peek()) จะมีตัวชี้ตำแหน่ง (Pointer) ไปยัง Node หัว (Head) และ Node ท้าย (Tail) อยู่แล้ว โปรแกรมจึงเข้าถึงและจัดการได้ในทันทีโดยไม่ต้องวนลูปวนหาข้อมูล ส่งผลให้มีระยะเวลาทำงานคงที่คงวาเสมอ
```text
=== ระบบคิวผู้ป่วย (Queue) ===
คิวเริ่มต้น: [P001, P002, P003, P004, P005]

เรียกผู้ป่วยเข้ารับบริการ 2 ท่าน...
ผู้ป่วยที่ได้รับบริการแล้ว: P001
ผู้ป่วยที่ได้รับบริการแล้ว: P002

มีผู้ป่วยใหม่ P006 และ P007 มาลงทะเบียนเพิ่ม...
ผู้ป่วยคนถัดไปที่ต้องเรียก: P003
จำนวนผู้ป่วยที่ยังรอคิวอยู่: 5 คน
สถานะของ Queue ปัจจุบัน: [P003, P004, P005, P006, P007]
```
